# Smolink Codebase Walkthrough

This is the living, line-by-line guide to the Smolink codebase. Read it alongside the source when learning the project or reviewing a change.

## Maintenance rule

Whenever a source, test, configuration, migration, Docker, CI, or frontend file changes, update this document in the **same change**:

1. Add or revise that file's section.
2. Explain each meaningful line or tightly related group of lines in order.
3. State what the code depends on, what it produces, and how it is verified.
4. Mark generated files as generated rather than trying to explain every generated line.

The implementation sequence is tracked separately in [backend-build-checklist.md](backend-build-checklist.md). Architecture decisions and teaching material live in the README and Engineering Playbook.

## `docker-compose.yml`

```yaml
services:
```

Begins the Compose service definitions. Compose creates one container for every service nested under this key.

```yaml
  postgres:
    image: postgres:16-alpine
```

Names the PostgreSQL service `postgres` and selects the official PostgreSQL 16 Alpine Linux image. Other Compose services will later be able to resolve `postgres` as a hostname; the current application connects through the host-mapped port instead.

```yaml
    environment:
      POSTGRES_USER: smolink
      POSTGRES_PASSWORD: smolink
      POSTGRES_DB: smolink
```

Provides the initialization values recognized by the official image. These credentials are intentionally local-development-only. The image creates the `smolink` user and `smolink` database on its first start.

```yaml
    ports:
      - "5432:5432"
```

Maps host port 5432 to Postgres's port 5432 inside the container. This matches the `localhost:5432` part of `DATABASE_URL` in `backend/.env`.

```yaml
    volumes:
      - postgres_data:/var/lib/postgresql/data
```

Mounts the named Docker volume `postgres_data` at Postgres's data directory. Container recreation does not discard the database.

```yaml
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U smolink -d smolink"]
      interval: 5s
      timeout: 5s
      retries: 5
```

Runs `pg_isready` inside the Linux Postgres container every five seconds. Compose marks the service healthy only when the database accepts connections for the configured local user and database; each attempt may take at most five seconds and five failed attempts mark it unhealthy.

```yaml
  redis:
    image: redis:7-alpine
    command: ["redis-server", "--appendonly", "yes"]
```

Names the Redis service, selects the official Redis 7 Alpine image, and starts Redis with append-only persistence enabled. Redis is still non-authoritative: durable URL data will belong to Postgres.

```yaml
    ports:
      - "6379:6379"
    volumes:
      - redis_data:/data
```

Maps host port 6379 to Redis and persists its local files in the named `redis_data` volume.

```yaml
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
```

Runs `redis-cli ping` directly inside the Linux Redis container. It needs no shell processing, so the `CMD` form is appropriate. The interval, timeout, and retries match the Postgres health check.

```yaml
volumes:
  postgres_data:
  redis_data:
```

Declares the two named Docker volumes referenced by the services.

## `backend/.env`

This ignored local file supplies the values read by `Settings`. Its `DATABASE_URL` targets the host-mapped Postgres port using `postgresql+asyncpg`; `REDIS_URL` targets Redis database 0; `JWT_SECRET` and `IP_HASH_SECRET` are distinct local-only secrets; `PUBLIC_BASE_URL` is `http://localhost:8000`; and `REDIS_CACHE_TTL_SECONDS` is `3600`. Keep real secrets out of this document and out of version control.

## Local infrastructure verification

The Postgres verification runs an async SQLAlchemy engine built from `get_settings().database_url`, opens one connection, executes `SELECT 1`, prints the scalar result, and disposes the engine in a `finally` block. On 2026-07-19 it returned `1`, proving that the application's configured connection string reaches the Compose Postgres service.

The Redis verification runs `docker compose exec redis redis-cli ping`. `exec` launches `redis-cli` inside the running `redis` service container, where `PING` returned `PONG` on 2026-07-19. These checks prove availability only; the application Redis client will be introduced in the rate-limiting and redirect-cache milestones.

## How the current application starts

```text
uv run fastapi dev app/main.py
  → imports app.main
  → module executes app = create_app()
  → FastAPI receives the application object
  → GET /health calls health()
  → FastAPI serializes the returned Python dictionary as JSON
```

## `backend/pyproject.toml`

```toml
[project]
```

Starts the standard Python project metadata table.

```toml
name = "backend"
```

Names the Python project. This is package metadata, not the FastAPI application title.

```toml
version = "0.1.0"
```

Sets the current package version. It can change independently of the API path version (`/api/v1`).

```toml
description = "Add your description here"
```

Provides package-description metadata. Replace this later with a useful Smolink description.

```toml
requires-python = ">=3.13"
```

Prevents installation on Python versions older than 3.13.

```toml
dependencies = [
    "fastapi[standard]>=0.139.0",
    "pydantic-settings>=2.14.2",
    "uvicorn[standard]>=0.51.0",
]
```

Declares runtime dependencies:

- `fastapi[standard]` supplies the web framework and its standard optional runtime pieces.
- `pydantic-settings` loads and validates environment-based settings.
- `uvicorn[standard]` runs the ASGI application locally and in production.

```toml
[tool.fastapi]
entrypoint = "app.main:app"
```

Configures the FastAPI CLI. `app.main:app` means “import the variable named `app` from `app/main.py`.”

```toml
[dependency-groups]
dev = [
    "httpx>=0.28.1",
    "pytest>=9.1.1",
]
```

Declares development-only dependencies:

- `httpx` supports FastAPI's in-process test client.
- `pytest` discovers and runs functions whose names begin with `test_`.

## `backend/app/main.py`

```python
from fastapi import FastAPI
```

Imports FastAPI, the class used to create the web application and register HTTP routes.

```python
def create_app() -> FastAPI:
```

Defines an application factory. It returns a `FastAPI` instance instead of constructing everything directly at module level. This makes later test configuration and startup resources easier to control.

```python
    app = FastAPI(
        title="smolink",
        version="0.1.0",
    )
```

Creates the application. `title` and `version` appear in generated OpenAPI documentation; they do not change API behavior.

```python
    @app.get("/health")
```

Registers the next function as the handler for HTTP `GET` requests to `/health`.

```python
    async def health() -> dict[str, str]:
```

Defines the route handler. `async` lets FastAPI await future I/O without blocking the server event loop. `dict[str, str]` documents that the handler returns string keys and values.

```python
        return {"status": "ok"}
```

Returns a Python dictionary. FastAPI serializes it to the JSON response `{"status":"ok"}` with status `200` by default.

```python
    return app
```

Returns the fully configured application from the factory.

```python
app = create_app()
```

Calls the factory and exposes the application object named `app`. Uvicorn and the FastAPI CLI import this exact variable.

## `backend/app/core/config.py`

```python
from functools import lru_cache
```

Imports Python's least-recently-used cache decorator. It will cache the settings object after it is first created.

```python
from pydantic_settings import BaseSettings, SettingsConfigDict
```

Imports Pydantic's settings base class and its configuration object.

```python
class Settings(BaseSettings):
```

Defines the application's typed settings schema. `BaseSettings` reads fields from environment variables and the configured `.env` file.

```python
    database_url: str
    redis_url: str
    jwt_secret: str
    ip_hash_secret: str
    public_base_url: str
```

Defines required string settings. Pydantic maps these to the uppercase environment names `DATABASE_URL`, `REDIS_URL`, `JWT_SECRET`, `IP_HASH_SECRET`, and `PUBLIC_BASE_URL`.

```python
    redis_cache_ttl_seconds: int = 3600
```

Defines an integer cache-TTL setting. It uses `REDIS_CACHE_TTL_SECONDS` when provided and otherwise defaults to 3,600 seconds.

```python
    model_config = SettingsConfigDict(
        env_file=".env",
        env_file_encoding="utf-8",
        extra="ignore",
    )
```

Configures settings loading:

- `env_file=".env"` loads local development values from `backend/.env`.
- `env_file_encoding="utf-8"` defines how that file is read.
- `extra="ignore"` permits unrelated variables in `.env` without rejecting startup.

```python
@lru_cache
def get_settings() -> Settings:
    return Settings()
```

Creates one cached settings instance. The first call validates and loads configuration; later calls return the same object. This avoids rereading environment state for every request.

## `backend/app/db/session.py`

```python
from collections.abc import AsyncGenerator
```

Imports the type annotation for an asynchronous generator. `get_session()` yields a session once and then performs cleanup.

```python
from sqlalchemy.ext.asyncio import AsyncSession, async_sessionmaker, create_async_engine
```

Imports SQLAlchemy's async session type, its factory constructor, and its async database-engine constructor.

```python
from app.core.config import get_settings
```

Imports the central typed configuration getter. Database connection details therefore remain in the local environment rather than Python source.

```python
settings = get_settings()
```

Loads the cached application settings once when this module is imported.

```python
engine = create_async_engine(
    settings.database_url,
    pool_pre_ping=True,
)
```

Creates the one shared asynchronous SQLAlchemy engine and its connection pool. `database_url` selects the `asyncpg` PostgreSQL driver. `pool_pre_ping=True` validates a pooled connection before reusing it, which makes local database restarts less likely to cause stale-connection failures.

```python
async_session_factory = async_sessionmaker(
    engine,
    expire_on_commit=False,
)
```

Builds the single reusable factory that creates `AsyncSession` objects from the engine. `expire_on_commit=False` keeps already-loaded ORM attributes available after a future transaction commits.

```python
async def get_session() -> AsyncGenerator[AsyncSession, None]:
    async with async_session_factory() as session:
        yield session
```

Defines the future FastAPI dependency. It opens a session, yields it to an endpoint or test, and closes it reliably when the consumer finishes or raises an exception.

## `backend/tests/test_db_session.py`

The smoke test imports `app.db.session`, asynchronously iterates over `session.get_session()`, runs `SELECT 1` through the yielded session, and asserts the returned scalar is `1`. It first failed because `get_session` did not exist; after implementation it passed against the Compose Postgres service on 2026-07-20.

## `backend/app/db/base.py`

```python
from sqlalchemy.orm import DeclarativeBase
```

Imports SQLAlchemy's modern declarative-model base type.

```python
class Base(DeclarativeBase):
    pass
```

Defines Smolink's shared ORM base. Future models inherit from `Base`, which gives SQLAlchemy one common metadata collection. Alembic will use that metadata collection to detect tables and generate migrations. `pass` is appropriate because the base has no project-specific behavior yet.

## Alembic migration environment

`alembic.ini` keeps Alembic's generated settings and script location. Its placeholder `sqlalchemy.url` is intentionally not replaced with a real connection string.

`alembic/env.py` imports `asyncio`, Alembic's `context`, SQLAlchemy's async engine support and `NullPool`, plus Smolink's `get_settings` and `Base`. It assigns `get_settings().database_url` to Alembic at runtime, so local credentials stay in `backend/.env`; it assigns `Base.metadata` as `target_metadata`, so future model tables are visible to Alembic; and it runs online migrations through an async engine and `connection.run_sync(...)`, which adapts Alembic's synchronous migration operations to the async connection. `uv run alembic current` connected to Postgres successfully on 2026-07-20 and found no revision, as expected before the first migration.

## `backend/.env.example`

```dotenv
DATABASE_URL=postgresql+asyncpg://smolink:smolink@localhost:5432/smolink
```

Documents the PostgreSQL connection string the future async SQLAlchemy engine will use. The values are safe local-development placeholders.

```dotenv
REDIS_URL=redis://localhost:6379/0
```

Documents the Redis instance, port, and logical database number used for local development.

```dotenv
JWT_SECRET=replace-with-a-long-random-secret
IP_HASH_SECRET=replace-with-a-different-long-random-secret
```

Documents two distinct secrets. The JWT secret signs access tokens; the IP-hash secret creates privacy-preserving keyed IP hashes. Replace both in the uncommitted `.env` file.

```dotenv
PUBLIC_BASE_URL=http://localhost:8000
```

Sets the public base URL used later to form returned short URLs and QR-code content.

```dotenv
REDIS_CACHE_TTL_SECONDS=3600
```

Sets the default non-expiring redirect-cache lifetime to one hour.

def test_health_returns_ok() -> None:

### `backend/tests/test_config.py`

```python
def test_settings_reads_environment_variables(monkeypatch) -> None:
```

Defines a configuration test. Pytest supplies `monkeypatch`, which safely changes environment variables for only this test.

```python
    from app.core.config import Settings
```

Imports the settings class inside the test. This keeps the dependency local to the behavior being tested.

```python
    monkeypatch.setenv("DATABASE_URL", "...")
    monkeypatch.setenv("REDIS_URL", "...")
    monkeypatch.setenv("JWT_SECRET", "...")
    monkeypatch.setenv("IP_HASH_SECRET", "...")
    monkeypatch.setenv("PUBLIC_BASE_URL", "...")
```

Supplies all required setting values using isolated test data. `monkeypatch` restores the real environment automatically afterward.

```python
    settings = Settings(_env_file=None)
```

Creates settings while deliberately ignoring a developer's local `.env` file. This makes the test repeatable on every machine and in CI.

```python
    assert settings.database_url == ...
    assert settings.redis_url == ...
    assert settings.jwt_secret == ...
    assert settings.ip_hash_secret == ...
    assert settings.public_base_url == ...
```

Verifies each environment variable reaches the matching typed Python field.

```python
    assert settings.redis_cache_ttl_seconds == 3600
```

Verifies the default applies when `REDIS_CACHE_TTL_SECONDS` is absent.
