# Initial data model design

**Status:** approved for implementation planning  
**Date:** 2026-07-20

## Purpose

Define the first durable PostgreSQL schema for Smolink. It must support guest
and owned URLs, efficient redirect lookup, synchronous first-release analytics,
and future authentication without making Redis responsible for durable data.

## Scope

This design covers SQLAlchemy models and the first Alembic migration only. It
does not add endpoint handlers, authentication flows, rate limiting, QR codes,
redirect caching, or analytics reporting.

## Model layout

Keep the shared SQLAlchemy declarative base in `app/db/base.py`. Put each table
in its own focused module:

```text
app/models/
├── __init__.py
├── user.py
├── url.py
└── click_event.py
```

`app/models/__init__.py` imports every model so Alembic sees the complete
metadata graph when it imports `app.db.base`.

## Tables

### User

| Field | Rules |
| --- | --- |
| `id` | `BIGINT` primary key; populated later by the Snowflake generator. |
| `email` | Required, unique, and indexed. Authentication code will normalize it with `strip().lower()` before persistence and lookup. |
| `password_hash` | Required Argon2 hash; never the plaintext password. |
| `created_at` | Required timezone-aware UTC timestamp generated by PostgreSQL. |
| `updated_at` | Required timezone-aware UTC timestamp generated by PostgreSQL and refreshed on update. |

### Url

| Field | Rules |
| --- | --- |
| `id` | `BIGINT` primary key; populated later by the Snowflake generator. |
| `short_code` | Required, globally unique, indexed code used by redirects. It stores both generated Base62 codes and custom aliases. |
| `destination` | Required destination URL string. |
| `owner_id` | Nullable foreign key to `users.id`; deleting a user sets this field to `NULL`, so their links keep working as guest links. |
| `expires_at` | Nullable timezone-aware UTC timestamp. |
| `total_clicks` | Required integer aggregate, defaulting to `0`. |
| `last_clicked_at` | Nullable timezone-aware UTC timestamp. |
| `created_at`, `updated_at` | Required timezone-aware UTC timestamps generated by PostgreSQL. |

There is deliberately no `custom_alias` column. Alias availability is checked
against `short_code`: validate and normalize the requested alias, look up that
value, then rely on the unique constraint as the final concurrency-safe guard.
This prevents a generated code from colliding with a custom alias.

### ClickEvent

| Field | Rules |
| --- | --- |
| `id` | Database-generated integer primary key. |
| `url_id` | Required foreign key to `urls.id`. Deleting a URL permanently deletes its events via `ON DELETE CASCADE`. |
| `clicked_at` | Required timezone-aware UTC timestamp generated by PostgreSQL. |
| `browser`, `os`, `device` | Required derived analytics dimensions. |
| `referrer` | Nullable referrer value. |
| `ip_hash` | Required keyed hash of the client IP. Raw IP addresses must never be stored. |

Click events are append-only facts, so they do not need `created_at` or
`updated_at`. The v1 schema also does not retain raw user-agent text; the
derived fields meet the first-release reporting requirement with a smaller
privacy footprint.

## Indexes and relationships

- `users.email` is unique and indexed for account lookup.
- `urls.short_code` is unique and indexed for the redirect hot path.
- `urls.owner_id` is indexed for a user's URL listing.
- `click_events` has a composite index on `(url_id, clicked_at)` for date-range
  analytics per URL.
- `Url.owner_id -> User.id` uses `ON DELETE SET NULL`.
- `ClickEvent.url_id -> Url.id` uses `ON DELETE CASCADE`.

## Timestamp policy

Use PostgreSQL timezone-aware UTC timestamps with database defaults. This gives
all writers the same clock and avoids application-server timezone drift.
`updated_at` uses a database-side update expression in the SQLAlchemy model;
the initial migration must reflect both creation and update behavior.

## Migration and verification

Generate one initial Alembic migration from the models, inspect it before
applying it, and apply it to the local Compose PostgreSQL service. Add focused
integration tests for the unique constraints and both foreign-key deletion
rules. The implementation follows the existing red-to-green workflow.

## Explicit deferrals

- Snowflake generation and Base62/alias validation belong to the next
  short-code-utilities milestone.
- Password-protected links would require a documented API and a migration, so
  they are not included in this initial schema.
- Soft deletion and restoration are future product features; v1 permanently
  deletes URLs and cascades their click events.
- Redis remains non-authoritative for URL data; its cache and rate-limit
  counters are outside this database schema.
